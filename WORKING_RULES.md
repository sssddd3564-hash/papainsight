# 파파인사이트 작업 기본 프롬프트

이 문서는 파파인사이트를 다른 PC나 다른 세션에서 이어서 작업할 때 반드시 먼저 확인하는 기본 작업 규칙입니다.

## 기본 원칙

- 사용자는 파파컴퍼니 ERP 홈페이지인 파파인사이트를 직접 운영/관리한다.
- 현재 목표는 거대한 ERP가 아니라 1차 MVP를 안정적으로 완성하는 것이다.
- GitHub 저장소는 `sssddd3564-hash/papainsight`를 기준으로 소통하고 관리한다.
- 기능은 작게 만들고, 확인하고, 기록한 뒤 다음 단계로 확장한다.
- 기존 작업물을 지울 때는 먼저 백업한다.
- 사용자가 코드 전체를 읽지 않아도 진행할 수 있도록 명령어는 복사/실행 가능한 형태로 제공한다.

## 현재 MVP

- 대시보드 1: 영업자료
- 대시보드 2: 거래처 관리
- 직원별 로그인 구조 예정
- 현재는 정적 HTML/CSS/JavaScript 기반 시제품
- 현재 직원 로그인 계정은 MVP 테스트용으로 프론트엔드에 임시 보관되어 있다. 운영 전에는 반드시 백엔드 인증과 DB 기반 사용자 관리로 교체한다.

## 매 작업 후 필수 확인

- 로컬 서버 실행 확인
- 주요 화면 직접 확인
- 로그인 동작 확인
- 메뉴 전환 확인
- 콘솔 또는 터미널 오류 확인
- 운영 배포 시 도메인 접속 확인

## E2E 검증 원칙

매번 중요한 수정 후 아래 흐름을 확인한다.

1. `http://localhost:4175/` 접속
2. `papa.admin` / `1234` 로그인
3. 영업자료 화면 표시 확인
4. 거래처 관리 화면 전환 확인
5. 로그아웃 확인
6. 배포 작업 후 `https://papainsight.co.kr/` 접속 확인

## 자동 백업 원칙

- 작업 중 10분 단위로 백업 상태를 점검한다.
- 큰 수정 전에는 압축 백업 또는 Git 커밋을 만든다.
- GitHub 저장소가 연결되면 작업 단위마다 커밋한다.
- 배포 전에는 운영용 압축 파일을 새로 만든다.

## 문서 최신화 원칙

컨텍스트가 길어져 압축되기 전, 또는 작업 흐름이 바뀌기 전에는 아래 문서를 최신화한다.

- `README.md`
- `CHANGELOG.md`
- `ERROR_LOG.md`
- `BACKUP_GUIDE.md`
- `WORKING_RULES.md`

## 오류 기록 원칙

오류가 나오면 그냥 넘기지 않는다.

- 어떤 명령어에서 발생했는지 기록
- 오류 메시지 요약
- 원인
- 해결 방법
- 다음부터 피할 점

같은 실수를 반복하지 않도록 `ERROR_LOG.md`에 남긴다.

## 사용자 안내 방식

- 어려운 설명보다 바로 실행 가능한 명령어를 우선한다.
- 서버 작업은 삭제보다 백업을 먼저 권장한다.
- 사용자가 PowerShell인지 SSH 서버 터미널인지 헷갈릴 수 있으므로, 명령어마다 실행 위치를 명확히 적는다.
- 민감 정보는 문서에 저장하지 않는다.
