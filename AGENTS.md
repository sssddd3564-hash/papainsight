# Codex 기본 작업 수칙

이 문서는 파파인사이트 작업에서 Codex가 먼저 확인하고 따라야 하는 기본 규칙입니다.

## 1. 문서 최신화

- 작업 중 변경된 기획, 구조, 기능, 결정사항은 관련 `MD`(엠디: Markdown 문서) 파일에 주기적으로 최신화한다.
- 컨텍스트(콘텍스트: 대화/작업 기억)가 길어져 압축될 가능성이 있으면, 압축 전에 반드시 문서를 최신화한다.
- 기본 확인 문서:
  - `AGENTS.md`(에이전츠 엠디: AI 작업 규칙 문서)
  - `PROJECT_NOTES.md`(프로젝트 노츠 엠디: 기획과 확정 내용 문서)
  - `CHANGELOG.md`(체인지로그 엠디: 작업 변경 이력 문서)
  - `README.md`(리드미 엠디: 실행 방법 안내 문서)
  - `ERROR_LOG.md`(에러 로그 엠디: 오류와 해결 내용을 기록하는 문서)
  - `WORKING_RULES.md`(워킹 룰스 엠디: 운영/협업 규칙 문서)

## 2. 구현 후 검증

- 기능 구현 후에는 직접 실행하고 테스트한다.
- 화면 작업은 브라우저에서 직접 확인한다.
- 가능하면 `E2E`(이투이: 처음부터 끝까지 사용자 흐름을 검증하는 테스트)를 진행한다.
- `DB`(디비: 데이터를 저장하고 관리하는 데이터베이스), 백엔드, 로그인, 결제, 관리자 기능처럼 실제 데이터와 연결되는 기능은 임의로 검증하지 말고 필요한 정보가 없으면 먼저 질문한다.

## 3. 오류 기록

- 실수, 오해, 잘못된 기억, 실패한 명령, 반복된 오류는 반드시 `ERROR_LOG.md`(에러 로그 엠디: 오류와 해결 내용을 기록하는 문서)에 남긴다.
- 오류 수정 후에는 원인과 수정 내용을 함께 기록해 같은 문제가 반복되지 않게 한다.

## 4. 개발 용어 표기

- 영어 개발 용어를 사용할 때는 한글 발음과 짧은 설명을 함께 적는다.
- 예시:
  - `API`(에이피아이: 프로그램끼리 데이터를 주고받는 연결 방식)
  - `DB`(디비: 데이터를 저장하고 관리하는 데이터베이스)
  - `E2E`(이투이: 사용자 흐름 전체 검증)
  - `CSS`(씨에스에스: 화면 스타일 코드)
  - `HTML`(에이치티엠엘: 화면 구조 코드)
  - `GitHub`(깃허브: 코드 저장소 서비스)

## 5. 브라우저 확인

- 오른쪽 브라우저를 보여줄 때는 가능한 경우 크기 조절/반응형 미리보기 툴바를 함께 적용한다.
- 데스크톱, 태블릿, 모바일 화면에서 주요 UI가 깨지지 않는지 확인한다.

## 6. 백업과 저장

- 기능과 `DB`(디비: 데이터를 저장하고 관리하는 데이터베이스) 관련 작업은 10분 단위로 백업하거나, 최소한 변경 내용을 문서와 변경 이력에 저장한다.
- 대화가 3회 이상 이어지고 작업이 누적되면, 현재 작업 상태를 `GitHub`(깃허브: 코드 저장소 서비스)에 보관할 수 있도록 사용자에게 저장 여부를 확인한다.
- `GitHub` 저장, 커밋, 푸시는 사용자 승인 없이 임의로 진행하지 않는다.

## 7. 보안 점검

- 배포, 로그인, 관리자, `API`(에이피아이: 프로그램끼리 데이터를 주고받는 연결 방식), `DB`(디비: 데이터를 저장하고 관리하는 데이터베이스) 작업 후에는 보안 점검을 진행한다.
- 비밀번호, 토큰, 개인정보, 관리자 계정 정보가 코드에 노출되지 않도록 확인한다.
- 실제 서비스 도메인과 연결된 작업은 캐시, 권한, 접근 제한, 공개 범위를 함께 확인한다.

## 8. 다른 세션에 복사할 주의 문구

- 이 프로젝트는 파파인사이트 작업이며, 이전 작업 맥락은 `MD`(엠디: Markdown 문서) 파일을 기준으로 이어받는다.
- 작업 전 반드시 `AGENTS.md`, `PROJECT_NOTES.md`, `CHANGELOG.md`, `README.md`를 먼저 확인한다.
- 기존 사용자가 만든 변경사항은 절대 임의로 되돌리지 않는다.
- 불확실한 내용은 추측으로 진행하지 말고 먼저 질문한다.
- 작업 후에는 구현 내용, 테스트 결과, 남은 위험 요소를 간단히 보고한다.
